# AI Interaction Guidelines

EduLink – School Communication & Academic Tracking SaaS

## Communication

Be concise, structured, and implementation-focused
Explain only non-obvious architectural decisions briefly
Always prioritize education domain correctness (student/parent/school relationships)
Ask for confirmation before:
major refactors
database schema changes
authentication or RBAC modifications
multi-tenant architecture changes
Do NOT introduce features outside the approved project scope
Never delete or modify files without explicit clarification
Maintain strict school data isolation (multi-tenant safety rule)

## Workflow

This is the standard workflow for every feature, fix, or enhancement:

1. 📄 Document
   Define feature in:
   @context/current-feature.md
   Include:
   scope
   affected modules
   database impact (if any)
   UI impact
2. 🌿 Branch
   Create feature branch:
   feature/[feature-name]
   or fix/[issue-name]
3. ⚙️ Implement
   Implement only what is defined in:
   current-feature.md
   Follow:
   coding standards
   Prisma schema rules
   RBAC constraints
4. 🧪 Test
   Verify in browser or simulator
   Run:
   npm run build
   Fix all build errors before proceeding
5. 🔁 Iterate
   Refine based on:
   UI correctness
   API behavior
   role-based access
   multi-tenant isolation
6. ✅ Commit
   Only commit when:
   build passes
   feature is fully working
   no console errors exist
7. 🔀 Merge
   Merge into main only after approval
8. 🧹 Delete Branch
   Delete feature branch after successful merge
9. 🔍 Review
   Periodically review AI-generated code for:
   correctness
   security
   performance
   consistency with existing architecture
10. 📌 Update Context
    Mark feature as completed in:
    @context/current-feature.md
    Move to feature history log

## Branching Strategy

Each feature must have a dedicated branch
Naming convention:
feature/attendance-module
feature/real-time-chat
fix/message-read-status
Rules:
One feature per branch only
No mixing unrelated changes
Always confirm branch deletion after merge

## Commit Standards

Ask before committing (never auto-commit)
Use conventional commits only:
Examples:
feat: add attendance marking system
fix: resolve message duplication in chat
chore: update prisma schema
refactor: simplify parent-student relation logic
Commit Rules:
One logical change per commit
Never include irrelevant changes
Never include messages like:
“Generated by AI”
“Claude assisted”
Keep history clean and readable

## When Stuck

If a problem persists after 2–3 attempts:
STOP implementation immediately
Document the issue clearly
Request clarification before proceeding
Do NOT:
attempt random fixes
change architecture without approval
bypass type or build errors
Always:
identify root cause first
confirm expected behavior
validate assumptions (especially in education data flow)

## Code Change Principles

Make minimal necessary changes only
Do NOT refactor unrelated modules
Preserve existing architecture patterns
Avoid “cleaning up” unless explicitly requested
Strict Rules:
No feature creep
No UI enhancements outside scope
No backend restructuring without approval
Maintain backward compatibility for all APIs

## Code Review Guidelines

AI-generated and human code must be reviewed for:

🔐 Security
Role-based access enforcement (critical)
No unauthorized cross-school data access
Input validation using Zod
Authentication checks on all endpoints
⚡ Performance
Avoid unnecessary re-renders (React Native / React)
Prevent N+1 queries in Prisma
Optimize real-time message handling
Cache repeated queries when needed
🧠 Logic Correctness
Validate:
attendance logic
parent-student mapping
grading calculations
Handle edge cases:
multiple children per parent
multiple schools per parent
missing attendance records
🧩 Architecture Consistency
Follow:
modular folder structure
server-first design
server actions pattern
Ensure consistency with:
multi-tenant architecture rules
RBAC system
Prisma schema conventions

## AI Usage Boundaries

AI can assist with:

code generation
refactoring (when approved)
API scaffolding
UI component creation
documentation generation
AI MUST NOT:
modify database schema without approval
introduce new modules independently
bypass workflow steps
change system architecture
override business rules (school/parent/teacher logic)
🚀 Core Principle

“Every change must preserve data integrity, role safety, and multi-school isolation.”
